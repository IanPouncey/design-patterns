Button examples

The buttons on this page have been created using <span> elements, with scripting to provide the expected behaviour. At first glance both buttons appear to be the same, but there are fundamental differences between the two design patterns.

Button 1

Button 1

Button 1 code


<span id="button1" class="default" onclick="javascript:toggleButton1(event);">Button 1</span>

Button 1 notes

The <span> element is semantically neutral. It has no implicit role that informs assistive technologies what the control's purpose is. In other words, there is nothing to indicate to assistive technologies that Button 1 is a button. It is simply treated as plain text.

The <span> element is not a natively interactive element, so Button 1 can't be focused on with a keyboard or targeted with speech recognition tools.

Button 1 can be focused on with a screen reader (using its virtual cursor (or equivalent)), but it cannot be activated with the keyboard. The enter key does nothing, and the space key fires its default action - scrolling the page.

The change in state of Button 1 is purely cosmetic. There is no programmatic indication, meaning that screen reader users remain oblivious to the button's current state.

Button 2

Button 2

Button 2 code


<span id="button2" tabindex="0" role="button" aria-pressed="false" onclick="javascript:toggleButton2(event);" onkeyup="handleBtnKeyUp(event);">Button 2</span>

Button 2 notes

Uses the ARIA role attribute with a value of button. This informs the browser that it should expose the Button 2 as a button through its accessibility API. This information is utilised by assistive technologies to inform users what they're dealing with.

Uses the HTML tabindex attribute with a value of 0. This places Button 2 into the tab order based on its location in the source order. The visual appearance of Button 2 now changes when focus moves to it.

Keyboard interaction for Button 2 is handled through scripting. Both enter and space key events are captured, and used to trigger the button's expected behaviour.

The aria-pressed attribute is used to indicate the state of Button 2. It is set to false initially, then scripted to toggle between true/false whenever the button is activated.

The visual appearance of Button 2 when pressed (or not) is hooked onto the aria-pressed attribute using an intelligent CSS selector, instead of a class name. Although this does not affect accessibility, it creates a cleaner separation between structure and design.

Based on the button design pattern from the ARIA Authoring Practices Guide.
