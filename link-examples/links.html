Link examples

The links on this page have been created using <span> elements, with scripting to provide the expected behaviour. At first glance both links appear to be the same, but there are fundamental differences between the two design patterns.

Note: When using HTML use the <a> element. Do not re-purpose other elements to create links.

Link 1

W3C website

Link 1 code


<span onclick="fetchResource1(event);">W3C website</span>

Link 1 notes

The <span> element is semantically neutral. It has no implicit role that informs assistive technologies what the element's purpose is. In other words there is nothing to inform assistive technologies that Link 1 is a link. It is simply treated as plain text.

The <span> element is not a natively interactive element. This means Link 1 cannot be focused on with the keyboard or targeted by speech recognition tools.

Link 1 can be focused on with a screen reader using its virtual cursor (or equivalent), but cannot be activated using the keyboard.

Link 2

W3C website

Link 2 code


<span tabindex="0" role="link" onclick="fetchResource2(event);" onkeyup="handleLinkKeyUp(event);">W3C website</span>

Link 2 notes

Uses the ARIA role attribute with a value of link. This informs the browser that it should expose Link 2 as a link through its accessibility API.

Uses the HTML tabindex attribute with a value of 0. This places Link 2 into the tab order based on its location in the source order. The visual appearance of Link 2 also changes when focus moves to it.

Keyboard interaction for Link 2 is provided through scripting. The enter key event is captured, and used to trigger the link's expected behaviour.

Based on the link design pattern from the ARIA Authoring Practices Guide.
