Button example

The buttons on this page have been created by re-purposing span elements, with scripting to provide the intended behaviour. At first glance they appear to be the same: both change appearance when hovered over with a mouse, and both change appearance when clicked on. That is where the similarities end however.

Button 1

Button 1

Button 1 notes

The span element is semantically neutral. It has no implicit role that informs assistive technologies what the control's purpose is. In other words, there is nothing to indicate to assistive technologies that Button 1 is a button. It is simply treated as plain text content.

The span element is not natively interactive, so Button 1 can't be focused on with a keyboard (or targeted with speech recognition tools).

Button 1 can be focused on with a screen reader (using the virtual cursor or equivalent), but it cannot be activated with the keyboard (whether a screen reader is running or not). The enter key does nothing, and the space key fires its default action - scrolling the page.

The change in state of Button 1 is purely cosmetic. There is no programmatic indication, meaning that screen reader users remain oblivious to the button's current state.

Button 2 notes

The ARIA role attribute is used to give Button 2 semantic purpose. The button value informs the browser that it should expose the control as a button through its accessibility API. This information is utilised by assistive technologies (like screen readers) to inform the user what they're dealing with.

The tabindex attribute is used to make Button 2 keyboard focusable. The 0 value places the button into the tab order based on its location within the source order. The visual appearance of Button 2 also changes when it receives keyboard focus. More information on using the tabindex attribute.

Keyboard interaction for Button 2 is handled through scripting. Both enter and space key events are captured, and used to trigger the button's expected behaviour.

The aria-pressed attribute is used to indicate the state of Button 2. It is set to false initially, then scripted to toggle between true/false whenever the button is activated.

The visual appearance of Button 2 when pressed (or not) is hooked onto the aria-pressed attribute using an intelligent CSS selector, instead of a class name. Although this does not affect accessibility, it creates a cleaner separation between structure and design.
